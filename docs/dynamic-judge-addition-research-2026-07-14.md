# 管理员端临时新增评委全链路调研

日期：2026-07-14  
范围：管理员账号维护、有效评分名册、派发、登录与会话、评分保存/提交、综合分、成绩展示、排名、审计、文件/MySQL 持久化、迁移恢复与测试。  
证据：仅使用当前仓库源码、脚本和测试；未修改业务代码或比赛数据。

## 结论摘要

当前系统已经具备“创建一个可登录的动态评委账号”的基础能力，但**不具备“开赛后临时增加一位有效评委并稳定完成后续评分”的完整能力**。

两个必须先解决的 P0 问题：

1. **赛中新增账号无法进入有效评分链路。** 首次派发会永久锁定全局有效名册，并为队伍保存快照；此后名册更新接口固定返回 409。新账号可以登录，但不在当前派发快照中，只能看到“当前队伍未分配给此评委”，而且也不会进入后续新队伍的快照。[账号创建](../server/contest-api-routes.mjs#L273) [名册锁定](../server/contest-api-routes.mjs#L411) [派发快照](../domain/contestControl.js#L121) [评委等待态](../src/App.jsx#L1242)
2. **现有灾备导出不能独立恢复动态新增账号。** SQL 导出只包含队伍、名册、评分和控制状态，不包含账号、密码哈希；动态账号的内部 ID 又是随机 `j-*`。导入到新库时目标库缺少该账号，名册/评分外键会拒绝恢复。[随机账号 ID](../contest-server.mjs#L753) [导出表范围](../scripts/export-state-to-sql.mjs#L54) [导入账号预检](../scripts/import-state-to-mysql.mjs#L70) [外键](../scripts/mysql-schema.sql#L72)

综合分算法、排名和服务端会话主体没有固定为 7 人，具备扩展基础；真正需要调整的是**名册生效边界、操作原子性、历史快照口径、投影布局、备份恢复和回归测试**。

推荐采用：

- 每队评分继续以不可变的 `rosterSnapshot` 为最终计分口径。
- 新增评委默认“从下一次派发开始生效”，不追改已开始或已完成队伍。
- 使用版本化有效名册，而不是“首次派发后永久不可变”的单一名册。
- “创建账号并加入下一版名册”由一个服务端事务完成。
- 当前队确需临时补入/替换时，走独立应急流程，要求原因、版本冲突检查、草稿策略和完整审计，不能复用普通名册编辑。

## 已验证的现状

### 1. 账号创建、更新和登录

- 管理员可创建任意格式合法、用户名唯一的评委账号；服务端生成随机内部 ID、执行 scrypt 哈希，并只返回脱敏账号信息。[创建接口](../server/contest-api-routes.mjs#L273) [账号 ID](../contest-server.mjs#L753) [公开字段](../contest-server.mjs#L469)
- 新账号创建后可立即登录；登录按持久化账号查询，不依赖默认 `001` 至 `007` 常量。[登录](../server/session-api-routes.mjs#L46)
- 同一账号支持多个会话。停用或重置密码会递增 `authVersion`，旧会话在后续请求中失效。[账号更新](../server/contest-api-routes.mjs#L325) [会话复核](../server/auth-session.mjs#L180)
- 创建新账号要求密码 8 至 256 位，前端也设置 `minLength=8`；这与默认 `001` 至 `007` 使用三位同名密码的现场心智不一致。临时增加 `008` 时直接使用 `008` 会被拒绝，需要明确新账号凭据政策。[服务端校验](../server/contest-api-routes.mjs#L280) [前端表单](../src/App.jsx#L998) [默认凭据](../contest-server.mjs#L78)

### 2. 账号与有效名册是两次独立操作

- 创建账号只向 `accounts` 增加记录，不会加入 `judgeRoster.judgeIds`。[创建 mutation](../server/contest-api-routes.mjs#L288)
- 赛前必须由管理员再勾选新账号并保存有效名册；前端没有“创建并入册”的单次动作。[名册前端](../src/App.jsx#L844) [账号表单](../src/App.jsx#L989)
- 两次请求之间若操作遗漏、网络失败或进程退出，会留下“账号已创建、可登录、但永远不参与评分”的半完成状态。再次创建同用户名又会返回 409。
- 锁册后前端仍保留普通“新增评委”表单，成功提示也只有“评委账号已创建”，没有明确提示该账号不能参与本场当前或后续评分，现场很容易把“创建成功”误解为“增员成功”。[创建表单与提示](../src/App.jsx#L861)
- 锁册后，前端会禁用所有启用评委的状态选择，包括刚创建、从未进入有效名册的新账号；服务端实际只禁止停用已锁定名册中的成员。这个前后端不一致会让管理员无法在页面上停用误建的赛中新账号。[前端状态禁用](../src/App.jsx#L499) [服务端停用边界](../server/contest-api-routes.mjs#L366)

### 3. 首次派发后的锁定模型阻止赛中新增

- 名册接口在 `lockedAt` 存在或已有当前派发队伍时拒绝任何修改。[名册接口](../server/contest-api-routes.mjs#L411)
- 首次派发会锁定名册，将当时启用且入册的评委复制到队伍 `judgeRosterSnapshot` 和 `activeAssignment.rosterSnapshot`。[派发](../domain/contestControl.js#L121)
- 之后每支首次派发的队伍都从仍然不变的全局名册生成快照；新账号没有任何受支持入口进入该名册。
- 服务端仍会向新账号下发当前派发，但保存评分时会校验账号必须在当前快照中；前端也用同一条件禁止评分。[评委状态载荷](../contest-server.mjs#L534) [服务端写分校验](../domain/contestControl.js#L205) [前端写分校验](../src/App.jsx#L1671)

### 4. 每队快照和综合分算法可支持动态人数

- `getTeamRoster()` 优先使用队伍快照，综合分、提交数和终态均以该快照为准；重复 ID 会去重。[队伍名册](../domain/contestControl.js#L46) [综合分](../domain/contestControl.js#L56)
- 队伍只有在快照中所有评委提交后才为最终状态；因此将新评委追加入已开始队伍会改变完成条件，可能让已完成队重新变为等待提交。[终态判断](../domain/contestControl.js#L72)
- 去高去低算法对任意 `>=3` 个有效提交都可计算：排序、删除一条最低和一条最高，再平均剩余分数，没有固定 7 人或固定 5 人分母。[算法](../shared/scoringRules.js#L115)
- 平分极值时也只删除一条最高和一条最低，符合当前 `slice(1, -1)` 语义；该规则应在竞赛规程中确认。

### 5. 管理员、成绩展示和排名的传播情况

- 排名和成绩展示都调用同一 `getCompositeSummary()`，提交数和名册数为动态值，没有服务端 7 人硬编码。[成绩展示载荷](../contest-server.mjs#L761) [排名载荷](../contest-server.mjs#L807)
- 最终发布要求全名册提交；临时发布只要求至少 3 人提交。人数增加到 8 人或更多后，3 人即可产生一个去高低后仅剩 1 人的暂算分，是否允许需要明确运营规则。[发布校验](../domain/contestControl.js#L173)
- 管理员当前队提交矩阵遍历的是全局 `judgeRoster`，不是当前队的 `activeAssignment.rosterSnapshot` 或队伍快照。当前永久锁定模型下二者恰好相同；改为版本化名册后，这里会对旧队显示错误人员。[管理员矩阵](../src/App.jsx#L937)
- `publicTeam()` 不返回队伍快照，因此前端无法仅靠现有团队对象正确查看历史队的评委明细。[公开队伍](../contest-server.mjs#L485)
- 匿名分数数组和 React 卡片数量是动态的，但大屏 CSS 固定 `repeat(7, ...)`；第 8 人开始进入隐式第二行，固定 16:9 舞台存在拥挤和重叠风险。[动态卡片](../src/App.jsx#L366) [固定七列](../src/styles.css#L3553)

### 6. 持久化和并发基础

- 文件模式将完整状态（包括动态账号）写入临时文件后 rename，正常重启可恢复新账号。[文件写入](../server/storage/contest-storage.mjs#L60)
- MySQL 的账号表、唯一用户名、会话表、名册表和评分外键可以承载动态账号；账号和评分使用增量 upsert，而非每次重写完整比赛状态。[表结构](../server/storage/contest-storage.mjs#L184) [账号 upsert](../server/storage/contest-storage.mjs#L323)
- 进程内单一写队列会在真正写入前重新鉴权，能串行化当前单 Node 实例的写操作。[状态队列](../server/state-store.mjs#L20)
- 但是账号创建与名册替换是两个独立 mutation；MySQL 下分别提交，无法保证“账号建立且已入册”的业务原子性。[账号持久化分支](../server/storage/contest-storage.mjs#L681) [名册持久化分支](../server/storage/contest-storage.mjs#L697)
- 已锁定名册中的活跃评委不能停用，避免队伍永远等不到其提交。这项保护在引入替补后必须升级为显式“替换/豁免”，不能简单删除。[停用保护](../server/contest-api-routes.mjs#L366)
- 前端 `AccountRow` 只要发现“名册已锁定 + 角色是评委 + 账号为启用”就禁用状态选择器，没有判断该账号是否真的属于锁定名册。因此赛中刚创建但未入册的新账号，服务端允许停用，管理员界面却无法停用；错误账号会变成 UI 上难以处置的孤立账号。[账号状态控件](../src/App.jsx#L499)

### 7. 审计现状

- 创建账号记录 `accountId`、用户名和角色；名册更新记录前后 ID；写分审计包含目标评委/队伍、两位小数项目分、总分和变化项。[账号审计](../server/contest-api-routes.mjs#L313) [名册审计](../server/contest-api-routes.mjs#L458) [写分审计](../server/contest-api-routes.mjs#L534)
- 日志统一脱敏 password/token/authorization/hash，并追加结构化 JSONL。[脱敏](../contest-server.mjs#L551)
- “新增并入册”目前只能从两条独立日志人工拼接，没有共同的 operation/correlation ID，也没有“从哪一队开始生效”的审计字段。

### 8. 导出、导入与灾备恢复

- `normalizeCompetitionState()` 会读取动态账号 ID 来校验名册和评分，但返回的传输状态不含 `accounts`；摘要也不统计账号。[标准化](../scripts/competition-state-transfer.mjs#L195) [摘要](../scripts/competition-state-transfer.mjs#L212)
- SQL 导出不包含账号表，因而不包含动态账号 ID、密码哈希、状态、认证版本和修订号。[SQL 导出](../scripts/export-state-to-sql.mjs#L54)
- 导入工具明确保留目标库账号，只导入队伍、评分、名册和控制状态；目标库必须事先已有源状态引用的所有评委 ID。[导入](../scripts/import-state-to-mysql.mjs#L134)
- `--dry-run` 仅本地标准化和打印摘要，不连接目标库检查缺失账号；含动态评委的备份会 dry-run 成功，正式导入才失败。[dry-run](../scripts/import-state-to-mysql.mjs#L227)
- 日检只执行该离线 dry-run，因此也无法提前发现动态账号不可恢复。[日检](../scripts/daily-inspection.mjs#L288)
- 文件模式没有滚动备份、fsync 或自动恢复实现；单个 JSON 损坏会同时影响账号凭据和比赛状态。[文件读取错误](../server/storage/contest-storage.mjs#L793)

## 本次隔离 API 演练

本次另起临时文件存储和独立端口，未读取或修改正式 `data/contest-state.json`，完成了两组真实 HTTP 演练：

1. 首次派发前创建第 8 位评委、加入名册、派发 `GZ01`，8 位评委分别登录并提交 `93.00` 至 `100.00`。结果为 `rosterCount=8`、`submittedCount=8`、`isFinal=true`，去掉 `93.00` 和 `100.00` 后综合分为 `96.50`。这证明当前计分、登录、派发快照和文件持久化运行路径可处理 8 人，但前提是增员发生在首次派发前。
2. 首次派发并完成评分后再创建第 9 位评委，账号创建返回 201、登录成功；名册更新返回 409，评分写入返回 409。新账号无法进入当前或后续评分链路，与静态代码结论一致。

可重复反馈命令基线为 `npm test`、`npm run check:admin-control`，以及本次使用临时服务执行的 8 人 HTTP 断言脚本。临时服务和数据目录已在演练后关闭并清理。

## 风险分级

### P0：未处理前不能承诺“现场临时新增评委”

1. 首次派发后名册不可变，新评委永远不能参与当前或后续队伍。
2. 动态账号不进入官方导出物，无法在新 MySQL 环境独立恢复；名册和评分会因缺账号外键而失败。
3. 导入 dry-run 和日检无法发现上述恢复失败，形成“演练通过、正式恢复失败”的假安全。

### P1：可能造成比赛口径错误或现场卡死

1. 创建账号与加入名册非原子，容易留下半完成状态。
2. 若直接放开全局名册锁并追改当前/历史队，会改变最终条件和综合分口径，已完成队可能回退，已发布大屏可能继续显示旧结果。
3. 管理员矩阵使用全局名册；版本化后会错误显示当前或历史队的参与人员。
4. 8 人以上投影固定七列，未经验证可能溢出固定 16:9 舞台。
5. 替补场景若简单停用原评委，会与现有“锁定名册成员不可停用”保护冲突，或让队伍永久等待。
6. 锁册后仍可无提示创建无法参赛的账号；误建的新账号又因前端过度禁用而不能直接停用，现场状态容易被误判。
6. 前端将所有启用评委误当作锁定名册成员，导致赛中新建但未入册的账号无法从管理员界面停用或归档。

### P2：可运维性和可解释性不足

1. 新增账号密码规则与默认三位账号规则不同，现场容易误操作。
2. 审计需要拼接两个事件，缺少生效队伍/派发版本和统一 operation ID。
3. 临时发布固定 3 人门槛在大名册下代表性不足。
4. 文件模式无自动滚动备份；动态账号加入后影响范围更大。
5. 真实后台摘要硬编码“1 位管理员”，演示页面仍有多处 `3/7`、`7 位` 文案；虽然不影响正式计分，扩展账号体系时应清理。[真实摘要](../src/App.jsx#L997)

## 建议方案

### 1. 明确三种不同业务动作

不要把“新增账号”“加入后续评分”“当前队应急替补”做成一个含糊按钮：

1. **仅创建账号**：建立可登录账号，但状态明确显示“未加入本场有效名册”。
2. **创建并从下一队加入**：默认推荐；账号与新名册版本在同一事务提交，返回明确的生效边界。
3. **当前队应急补入/替换**：独立高风险操作；要求原因、影响预览和期望版本，不允许静默追改历史。

### 2. 版本化名册与不可变队伍快照

建议模型：

```text
judge_roster_versions
  roster_revision, status, effective_from_assignment_revision,
  created_by, reason, created_at

judge_roster_members
  roster_revision, account_id, sort_order, member_status

active_assignment
  team_id, assignment_revision, roster_revision, status

team_roster_snapshot / assignment_roster_snapshot
  team_id, assignment_revision, account_id, sort_order
```

规则：

- 每次普通增员创建新的名册版本，不修改旧版本。
- 下一次派发把当时生效的名册复制为队伍快照；综合分永远只读该快照。
- 已开始、已完成、已发布队伍保持原快照和原结果，不因后来增员变化。
- 当前队应急变更必须创建新的 `assignmentRevision` 和新快照，并明确旧草稿如何处理。

### 3. 当前队应急变更的安全边界

优先规则：

- 当前队尚无人产生有效草稿时，可允许“重新生成当前派发快照”，提升 `assignmentRevision`，旧请求统一 409。
- 已有人保存或提交后，默认只允许新评委从下一队生效。
- 若竞赛规则允许中途替补，必须要求管理员选择“新增”或“替换”，填写原因，展示已有提交影响；提交后撤下/标记已发布成绩为 `review_required`。
- 替换原评委时不能删除其历史评分；应在新快照中标记该成员对本次派发的状态，并记录被替换关系。
- 不能把旧评委已有分数自动复制给新评委。

### 4. 原子 API 和冲突控制

建议新增组合接口，而不是让前端连续调用两个旧接口：

```http
POST /api/judges
{
  "username": "008",
  "displayName": "评委 08",
  "password": "...",
  "enrollment": "next_assignment",
  "expectedRosterRevision": 3,
  "reason": "现场临时增补"
}
```

服务端在一个事务中完成：

1. 校验管理员会话、用户名唯一、密码策略和期望名册版本。
2. 创建账号。
3. 创建下一版名册及成员关系。
4. 写审计 outbox/结构化审计事件。
5. 返回账号、名册版本、生效范围和是否影响当前队。

使用幂等请求 ID，避免客户端超时后重试产生重复账号。版本冲突返回 409 和当前权威名册，不自动覆盖。

### 5. 前端管理员体验

- 新增表单必须显示选择项：“仅创建账号”或“从下一队加入有效名册”。首次派发前可默认后者。
- 开赛后不要继续显示一个看似成功但实际无法评分的普通“新增评委”流程。
- 成功提示必须包含：账号、是否入册、从哪一队/哪个派发版本生效。
- 账号列表增加状态：`未入册`、`下一队生效`、`当前队有效`、`历史有效/本轮不参与`。
- 当前队提交矩阵只使用 `activeAssignment.rosterSnapshot`；历史队详情由服务端返回该队快照。
- 账号状态控件的锁定条件必须同时检查 `judgeRoster.judgeIds.includes(account.id)`；未入册账号仍可停用，已入册账号通过替补流程处置。
- 密码采用统一赛事政策。若由系统生成一次性密码，只在创建成功响应中展示一次，审计和日志不得记录。

### 6. 投影、排名和临时发布

- 大屏卡片布局改为由人数驱动的受控布局，并至少验证 3、7、8、9、12 人在 1920x1080、1366x768 和非 16:9 letterbox 下的像素结果。
- 评委显示顺序继续匿名化，标签使用 `评委1...N`，不暴露账号身份。
- 排名继续按队伍自身快照计算，不按“当前最新名册”重算历史。
- 临时发布建议采用“至少 3 人且达到该队名册的一定比例”或仅管理员明确强制发布；具体比例需赛事规则确认。

### 7. 存储、导出和恢复

- 官方导出必须包含动态 `accounts`，至少覆盖内部 ID、用户名、显示名、角色、状态、密码哈希、password/auth/revision 版本和时间；不得导出会话 token。
- 导入必须在同一事务中按 `accounts -> teams -> roster versions/members -> entries -> controls` 顺序恢复，并校验所有 FK。
- SQL 导出和 JSON 转移摘要增加：账号总数、动态账号数、有效名册 ID、评分引用账号 ID、缺失账号数。
- 新增真正的只读 `preflight`：连接目标库但不写入，检查 schema 版本、目标账号冲突、缺失账号、FK、字符集和预计行数。
- 文件模式增加滚动备份、校验和和恢复演练；正式决赛仍以 MySQL 为事实源。

### 8. 审计事件

新增单一业务事件 `judge_enrollment_created`，至少包含：

- actor、operationId、accountId、username（不含密码）。
- previousRosterRevision、nextRosterRevision。
- enrollmentMode、effectiveAssignmentRevision、effectiveTeamId。
- currentAssignmentRevision、是否影响当前队。
- 若为替换：replacedAccountId、原因、已有草稿/提交处理结果。

评分审计继续保留现有精确两位小数、总分和变化项。

## 测试与验收清单

### 领域与 API

- 赛前“创建并入册”一次成功；账号、名册和审计同时存在。
- 账号创建成功但名册写失败时整体回滚；相同幂等 ID 重试不重复创建。
- 两位管理员以相同 roster revision 同时增员，只有一方成功，另一方 409。
- 首次派发后新增评委默认只进入下一次派发；当前队快照、草稿、提交数和综合分不变。
- 已完成队和已发布队不因新名册版本重算。
- 当前队无人评分时应急增员提升 assignment revision，旧设备在飞写入被拒绝。
- 当前队已有草稿/提交时普通增员被拒绝或只允许下一队生效。
- 替补流程保留原评委历史条目，不复制分数，并正确处理最终条件。
- 新评委登录后：生效前显示明确等待态，生效后自动进入正确队伍并能保存、提交。
- 名册锁定后创建一个未入册账号，管理员仍能停用/归档；锁定名册成员仍受保护。
- 停用、改密、登出与在飞保存竞态仍不能让旧会话写入。

### 计分、展示与排名

- 对 3、4、7、8、9、12 人验证去一条最高、去一条最低和四舍五入到分。
- 验证最高/最低并列时只移除一条，并与赛事规则一致。
- 每个队伍的最终条件只取自身快照；不同队伍允许不同 roster count。
- 管理员提交矩阵、队伍列表、投影和排名的 `submittedCount/rosterCount` 一致。
- 8+ 人投影无溢出、遮挡或字号不可读；匿名卡片全部可见。
- 临时发布门槛按最终确认规则执行，不因人数变化产生误导性暂算分。

### File 与 MySQL

- 文件模式：新增并入册、评分、重启后账号/名册/快照/条目/会话预期全部恢复。
- MySQL：动态账号创建、入册、派发、评分、服务重启后全部恢复。
- 组合事务任一步故障时无孤立账号、半份名册或悬空条目。
- 从包含动态账号的 file 状态导出到空白 MySQL，账号、密码哈希关系、名册和评分完整恢复。
- `--dry-run/preflight` 在目标缺动态账号、schema 旧版或 FK 不满足时提前失败。
- 导出摘要和恢复后摘要逐项匹配账号数、名册数、条目数、提交数和控制版本。
- 账号改密后旧会话失效，新密码在恢复后可登录；日志和导出过程不泄露明文密码/token。

### 审计与运维

- `account_create`、名册生效和应急替换均有 actor、operationId、生效范围和原因。
- JSONL 失败时关键数据库事务仍可通过 outbox 补写，而不是永久丢审计。
- 赛事手册增加临时新增/替补操作步骤、密码交付、失败回滚和恢复演练。
- 决赛前至少完成一次“比赛中新增第 8 位评委 -> 下一队评分 -> 投影 -> 排名 -> 服务重启 -> 灾备恢复”的全流程演练。
