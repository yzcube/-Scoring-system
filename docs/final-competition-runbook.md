# 决赛评分现场 Runbook

更新日期：2026-07-14

## 目标

按本场开赛配置准备评委平板，标准评审团为 7 台；有特邀评委时按实际人数增加设备。管理员设备和投屏控制设备访问同一个局域网评分服务器。管理员是唯一的队伍派发和成绩发布操作者；评委只对当前被派发队伍评分。

## 赛前准备

1. 主机、管理员设备、评委平板接入同一个受控网络，关闭主机休眠和自动更新重启。
2. 正式云服务器使用宿主机 MySQL 8.0，单实例运行 Node 服务。不要增加数据库 Docker 容器。
3. 按 [mysql-deployment.md](./mysql-deployment.md) 完成首次初始化、MySQL smoke 测试和备份演练。
4. 记录服务启动日志的 `LAN access` URL。评委平板不要使用 `127.0.0.1`。
5. 初始账号为评委 `001` 至 `007`，密码各自与账号相同；管理员为 `admin` / `admin123`。正式开放前重置管理员密码。
6. 同一账号可在多台设备登录，但同一评委同一评分条目会受版本冲突保护。备用设备接管前先确认主设备停止操作。

## 正式启动

```bash
npm ci
npm run build
CONTEST_STORAGE=mysql \
CONTEST_REQUIRE_ADMIN_PASSWORD_ROTATION=1 \
CONTEST_MYSQL_HOST=127.0.0.1 \
CONTEST_MYSQL_DATABASE=campus_final_scoring \
CONTEST_MYSQL_USER=contest_scoring \
CONTEST_MYSQL_PASSWORD='<正式密码>' \
REQUIRE_EMPTY_STATE=1 \
node contest-server.mjs
```

如果服务拒绝启动，先检查是否仍有彩排评分条目。不要直接删除队伍、账号、名册或控制表来绕过检查。

## 清场

正式赛清场只有以下一个流程，不能在运行中的服务下直接删除评分后继续使用：

1. 执行完整 MySQL 与 JSONL 审计日志备份。
2. 对彩排开启过的每个组执行“应急处置 → 重新配置当前组”。
3. 确认所有组为待配置、当前派发为空、投屏无成绩；按部署文档清除残余评分行。
4. 使用 `REQUIRE_EMPTY_STATE=1` 启动。若服务拒绝启动，按错误中的残留类别排查，不能删除队伍、账号、名册或控制表绕过。
5. 重新核对正式赛队伍与评委，保存并开启当天组别，再派发首支队伍。

## 赛前验收

1. 管理员登录；尚未开赛时默认进入“开赛配置”，已有组别开启时默认进入“赛事控制”。确认本场计划使用的评委账号均为启用状态。
2. 本场全部评委分别登录对应账号，确认未派发时只显示等待状态，没有组别、队伍列表或投屏入口。
3. 管理员在“队伍管理”确认出场顺序；若需要调整，先保存完整顺序。然后在“开赛配置”选择当天组别，确认本场队伍数和评委数，保存并开启该组。未使用组别保持待配置状态。
4. 管理员派发一支测试队伍。所有评委端应显示相同队名，点击评分输入区可打开两位小数数字键盘。
5. 按本场配置的全部评委各保存并提交测试分。标准评审团应显示 `7 / 7`；若主委会安排特邀评委，则以开赛配置人数 `N / N` 为准，并确认最终综合分。
6. 管理员在“成绩展示”发布最终成绩，打开投屏页确认只显示被发布队伍、匿名评委总分、综合分和去掉的最高/最低分。
7. 管理员撤回一份测试提交，确认投屏停止显示该旧最终成绩；重新提交后需要管理员显式再次发布。
8. 备份后，对彩排开启过的每个组执行“应急处置 → 重新配置当前组”，确认组别回到待配置、当前派发为空、投屏无成绩。仅在全部组完成后清理残余评分行，并用 `REQUIRE_EMPTY_STATE=1` 启动；若仍有赛次残留，服务会拒绝启动，不能绕过。

## 每队操作

1. 路演开始时，管理员在“赛事控制”选择该队并派发。
2. 评委录入每个项目的获评分数，可精确到 `0.01`。评分键盘只在选中评分项时出现；退格只删除一个字符，清空才会清除当前项草稿。
3. 评委提交后，管理员在提交矩阵确认状态。未完成有效名册提交时，正常派发下一队会被阻止。
4. 全部名册提交后，管理员确认最终综合分。综合分为去掉一个最高和一个最低评委总分后的平均值；同分仍各去掉一条记录。
5. 第 1、2 支队伍形成最终成绩后继续派发，不立即公布；第 3 支完成后，在第 4 支开始前进入“成绩展示”，按出场顺序依次发布前三支成绩。
6. 从第 4 支队伍起，每支队伍形成最终成绩后建议立即发布，再进入下一支。发布时间仍由管理员决定，系统不得自动发布或把该建议做成服务端硬门禁。

## 结束赛次

1. 最后一支有效队伍形成最终成绩后即可结束赛次。成绩是否发布由管理员决定，不构成关赛门禁；后台会同时保留“发布成绩”和“暂不展示，直接结束本组”两种操作。
2. 管理员点击“结束本组比赛”。系统会再次核对本组不存在未完成队伍，然后把组别改为已结束、释放当前组控制并关闭当前派发。
3. 确认顶部显示“本组已完成”，开赛配置中的组别状态为“已结束”，评委端已回到等待状态。
4. 若仍有未完成队伍或当前派发未完成，结束操作会被拒绝；不得通过直接修改数据库绕过。

## 应急操作

- **误提交**：管理员在当前队提交矩阵对对应评委点击“撤回提交”。评委修改后重新提交。
- **重评**：管理员在当前队提交矩阵对对应评委点击“清空重评”。这是直接应急操作，之后评委重新录入。
- **强制切队**：只在必须中断时使用。填写至少三个字符的原因，系统记录审计并拒绝随后到达的旧派发版本评分。
- **应急重新开赛**：在“开赛配置”选择当前组。尚未产生评分时可直接撤回开赛；已有评分时需连续两次确认，系统只清除该组评分和评委快照，保留队伍资料、报名编号、抽签顺序和账号。重新核对队伍数与评委数、保存配置并再次开赛后才能继续派发。
- **评委设备掉线**：不要刷新或清空页面。恢复网络后，系统只会在当前派发版本仍匹配时补交本地草稿；管理员确认提交矩阵。
- **临时增加评委**：在“评委管理”使用“从下一支队伍起加入评分”。创建成功后核对“计划人数”和“当前队快照人数”；当前队人数必须保持不变，新评委从下一支首次派发队伍开始评分。
- **当前队评委离场**：展开当前队提交矩阵，在离场评委一行选择“替换评委”，选择启用的替补评委并填写原因。若原席位已有评分，确认后只清除被替换席位当前队的评分；其他评委成绩、历史队伍和后续计划名册保持不变。
- **服务中断**：暂停录分。MySQL 模式检查数据库连接和审计日志后重启服务；管理员核对当前派发和提交矩阵，再继续。
- **设备损坏**：在备用设备的独立浏览器标签页登录同一账号。确认服务器中已有分数和当前队伍后继续。
- **管理员看板与平板不一致**：以管理员状态和服务端 `/api/state` 为准，确认设备均连接到同一 `LAN access` 地址。

## 备份与日志

每 3 至 5 支队伍或每轮结束执行一次备份：

```bash
mkdir -p backups
mysqldump --single-transaction --quick -uroot -p campus_final_scoring \
  contest_final_teams \
  contest_final_accounts \
  contest_final_account_sessions \
  contest_final_judge_roster \
  contest_final_entries \
  contest_final_control_state \
  contest_final_audit_events \
  > "backups/campus-final-$(date +%Y%m%d-%H%M%S).sql"
```

同时备份服务 JSONL 日志。日志包含关键审核事件和两位小数的评分变动，但不应包含密码、Bearer token 或密码哈希。

从灾备恢复后执行 `DELETE FROM contest_final_account_sessions;`，重启单实例服务并要求所有设备重新登录，再核对计划名册、当前派发快照和动态评委账号。

## 赛后

1. 停止服务后完成最后一次 MySQL 与 JSONL 备份。
2. 记录强制切队、撤回、清空或账号维护等应急审计事件。
3. 保留比赛数据；下一轮仅按“清场”删除条目，除非已明确批准完整赛事重置。
