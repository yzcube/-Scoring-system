## 审计范围

- 登录页：账号密码登录、错误提示、演示账号入口。

- 管理员端：评委提交进度、20 支队伍综合评分、少于 3 人提交状态、去最高去最低计算结果。
- 数据隔离：7 位评委账号、1 位管理员账号、localStorage/session 恢复、跨角色可见性。

## 实测通过项

- 登录页可以通过账号区分角色：评委进入评分端，管理员进入工作台。
- 错误账号密码会显示纯文本错误提示：`账号或密码不正确`。
- 评委端队伍列表包含 20 支队伍，并显示每队本评委评分状态。
- 评分项整行可点击，点击后打开数字键盘；键盘支持两位小数录入。
- 缺项提交会显示纯文本提示，并定位到第一个缺失评分项。
- 完整提交后关闭键盘、回到顶部，并显示纯文本提交提示。
- 评委 01 提交 A01 后，评委 02 登录同一队显示空白评分，UI 层面没有直接继承评委 01 分数。
- 管理员工作台显示 7 位评委进度和 20 支队伍结果。
- 综合分规则代码和运行结果一致：已提交总分排序后去掉最高和最低，再平均剩余分数。实测 80、90、100 三位评委提交时，A01 综合分显示 `90.00`。
- 少于 3 位评委提交时，管理员端显示 `至少 3 位评委提交后计算`，不提前计算综合分。

## 严重问题

1. 登录与角色控制完全在前端，且演示账号卡片可免密登录。
   - 位置：`src/App.jsx:17`、`src/App.jsx:848`
   - 影响：任何能打开页面的人都能看到全部账号密码，并点击管理员卡片直接进入后台。
   - 结论：适合演示原型，不适合真实比赛系统。

2. 会话恢复信任 `localStorage`，可伪造管理员身份。
   - 位置：`src/App.jsx:367`
   - 影响：手动设置 `campus-final-tablet-session-v1=admin` 后刷新即可进入管理员页面。

3. 评委数据和管理员综合分都存在同一个本地 `localStorage` 对象中。
   - 位置：`src/App.jsx:346`、`src/App.jsx:476`、`src/App.jsx:923`
   - 影响：浏览器用户可以篡改任意评委分数和提交状态；管理员端会信任这些本地数据计算综合分。

4. 历史阶段缺少共享数据源，管理员只能看到当前浏览器本地数据。当前版本已改为共享评分服务器。
   - 位置：`package.json` 仅有 Vite 前端脚本；状态从 `src/App.jsx:454` 本地初始化。
   - 影响：7 位评委若分别用不同平板，管理员端不会自动收到各设备评分。

## 高风险问题

1. 多标签页模拟时可能发生旧快照覆盖。
   - 位置：`src/App.jsx:488`
   - 影响：一个评委页面加载后，另一个评委提交；前一个页面再保存，可能用旧 `entriesByJudge` 覆盖后提交数据。

2. 数字键盘打开后缺少焦点管理。
   - 位置：`src/App.jsx:585`、`src/App.jsx:1089`
   - 影响：外接键盘或辅助技术用户很难直接进入数字键盘操作。

3. 窄屏提交栏存在挤压风险。
   - 位置：`src/styles.css:1125`
   - 影响：320px 宽度下，固定列宽和大号总分文本可能挤压中间总分显示。

## 中低风险问题

- 提交/更新提交没有提交历史、版本号、首次提交时间；管理员无法看出分数是否被改过。
- 队伍列表使用 `listbox/option` 语义，但没有方向键、Escape 关闭、焦点回收等配套键盘模型。
- 缺项提交和重置二次确认只靠短暂 toast，现场压力下容易漏看。
- 原问题：队伍列表中的“退出登录”按钮在列表底部且最小高度为 36px，低于常用触控基线。已调整为队伍列表顶部 sticky 区域，按钮最小高度 48px。
- 回归修复：登录和退出都会重置页面内会话 UI 状态，包括当前队伍、评分焦点、键盘草稿、队伍列表、数字键盘和重置二次确认状态。
- 回归修复：评委默认评分页新增固定可见的“退出登录”按钮，放在 compact hero 外部；不再需要先打开队伍列表才能退出。
- 回归修复：评分键盘打开和缺分提交定位后会接收 DOM 焦点；队伍列表移除未完整实现的 listbox/option 语义；管理员评委状态点增加逐评委可访问名称。
- 管理员页 7 个评委状态点主要依赖颜色和 `title`，个体提交状态对辅助技术不够清楚。
- 长队名多处单行省略，窄屏下可能丢失关键信息。

## 参考标准与资料

- OWASP Broken Access Control：访问控制应在可信服务端或服务端 API 中执行，不能信任客户端可修改的元数据。https://owasp.org/Top10/2021/A01_2021-Broken_Access_Control/
- OWASP Top 10 2025 仍将 Broken Access Control 列为 A01。https://owasp.org/Top10/2025/A01_2025-Broken_Access_Control/
- W3C Web Storage：`localStorage` 是按 origin 提供的本地存储区域，不是跨设备共享数据库。https://www.w3.org/TR/2016/REC-webstorage-20160419/
- WCAG 2.2 Target Size Minimum：指针目标至少应满足 24 x 24 CSS px 或符合例外条件。https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html
- W3C 建议使用最新 WCAG 版本作为网页可访问性标准。https://www.w3.org/WAI/standards-guidelines/wcag/

## 建议优先级

1. 正式比赛前必须接入服务端认证、服务端提交 API、服务端综合分计算和提交审计日志。
2. 移除页面上的演示账号卡片，至少管理员不能免密进入。
3. 保存分数时按 `judgeId/candidateId` 合并最新数据，避免同源多标签覆盖；正式系统应由后端原子写入。
4. 管理员端展示每位评委最近提交时间、更新次数和提交状态文本。
5. 修复数字键盘焦点管理、队伍列表键盘行为、窄屏提交栏布局、状态点可访问名称。
