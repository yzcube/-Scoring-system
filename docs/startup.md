# 启动说明

更新日期：2026-07-14

现场操作先阅读 [现场一分钟操作卡](./final-quick-guide.md)；全部正式文档入口见 [README](./README.md)。

## 本机开发与复盘

文件模式只用于本机开发、临时评审或复盘：

```bash
npm run contest
```

正式前需要清空本机文件模式评分时使用：

```bash
npm run contest:fresh
```

本机访问地址为 `http://127.0.0.1:8776/`。平板不能使用该地址；局域网设备必须使用服务启动日志输出的 `LAN access` 地址。

## 正式 MySQL 启动

正式云服务器使用宿主机 MySQL 8.0 和单个 Node 进程，不新增数据库 Docker 容器。Node 版本必须符合 `.nvmrc` 中的 `22.12.0`。

```bash
cd "<项目目录>"
npm ci
npm run build
CONTEST_STORAGE=mysql \
CONTEST_REQUIRE_ADMIN_PASSWORD_ROTATION=1 \
CONTEST_MYSQL_HOST=127.0.0.1 \
CONTEST_MYSQL_DATABASE=campus_final_scoring \
CONTEST_MYSQL_USER=contest_scoring \
CONTEST_MYSQL_PASSWORD='<正式密码>' \
REQUIRE_EMPTY_STATE=1 \
node contest-server.mjs
```

首次 MySQL 部署先按 [mysql-deployment.md](./mysql-deployment.md) 初始化表、80 支队伍、8 个账号和计划评分名册。不要手工写入密码哈希或评分数据。

## 设备入口

- 管理员、按本场配置数量准备的评委平板和投屏控制设备访问同一个 `LAN access` URL。标准评审团为 7 台；有特邀评委时增加对应设备。
- 每个浏览器标签页将认证令牌保存在 `sessionStorage`；管理员和评委可在备用设备的不同标签页同时保持独立会话。
- 评委 `001` 至 `007` 的初始密码与账号相同；管理员初始账号为 `admin`、密码为 `admin123`。登录页不预填凭据，也不显示账号卡。
- 管理员必须在正式开放网络前重置管理员密码，并限制现场网络访问范围。

## 正式前清场

先备份 MySQL 和 JSONL 审计日志，再对彩排开启过的每个组执行“应急处置 → 重新配置当前组”。确认所有组回到待配置、当前派发为空、投屏无成绩后，按 [mysql-deployment.md](./mysql-deployment.md) 清除残余评分行，并使用 `REQUIRE_EMPTY_STATE=1` 启动。门禁若报告组别、派发、投屏、名册锁或队伍快照残留，必须先排清，不能只删评分后继续运行。

文件模式清场前先复制 `data/contest-state.json`，再删除该文件；不要用删除本地文件的方式清理正式 MySQL 数据。

## 现场流程

1. 管理员登录后先进入“开赛配置”，选择当天比赛组别，逐项确认本场参赛队伍和评委；界面显示最终队伍数与评委数，保存后点击“开启本组比赛”。不同组别的配置分别保存，未开启组别不会参与当天派发。
2. 管理员进入“赛事控制”，从已开启组别的本场队伍中选择当前队伍并派发评分。评委端只显示这支被派发队伍；未派发时只能等待，不能录分或提交。每支队伍首次派发时形成独立评分快照。
3. 评委逐项使用两位小数键盘录分并提交。管理员确认提交矩阵达到该队评分快照的全部人数。
4. 成绩由管理员按现场节奏发布：通常前 3 支队伍完成后按出场顺序集中公布，第 4 支起建议每队形成最终综合分后立即发布。投屏页只读取管理员发布的结果，系统不自动发布。
5. 抽签或现场人员变化需要整体调整本组人数时，回到“开赛配置”。尚无评分可直接“撤回开赛并调整”；已有评分时使用“应急重新开赛”，并在 8 秒内第二次确认。该操作只清除当前组评分和队伍评委快照，保留队伍资料、报名编号、抽签顺序与账号，随后重新选择队伍和评委、保存并开赛。
6. 误提交时，管理员可以在当前队的提交矩阵中单击撤回或清空。已发布最终成绩会立即变为待复核，大屏不再显示旧分数。
7. 最后一队形成最终成绩后，管理员按顶部“结束本组比赛”显式关闭赛次。成绩发布是独立的投屏控制流程，不阻断关赛；只有关闭成功后，组别才进入已结束状态并释放当前组控制。

正常情况下，上一个队未完成全部有效评委提交时不能派发下一队。应急强制切换必须填写原因，并会记录审计。

临时增加评委时，在“评委管理”选择“从下一支队伍起加入评分”。系统会原子创建账号并更新计划评分名册；当前队和历史队伍人数不变。若当前队评委临时离场，应在当前队提交矩阵使用“替换评委”，填写原因并核对被清除的当前队席位；不要通过计划名册修改当前队。

## 审计与核对

审计日志默认写入 `data/logs/contest-server-YYYY-MM-DD.jsonl`。它记录登录/退出、队伍和账号维护、名册、派发、评分保存或提交、撤回、清空、冲突、发布以及服务错误；轮询和健康检查不会刷屏。

评分审计包含操作者、目标评委和队伍、精确到两位小数的项目分、总分及修改前后差异，不记录密码、Bearer token、Authorization 头或密码哈希。

启动后至少核对：

- 管理员能看到赛事控制、队伍管理、评委管理和成绩展示模块。
- 未派发时评委端显示等待；派发后所有评委端显示同一支队伍。
- 管理员能看到当前队评分快照内全部评委的提交进度，且最终综合分在该快照全部提交后形成。
- 投屏页不提供队伍选择或评分操作，只显示管理员发布的结果。
- 每个角色都有无需滚动到队伍列表底部的退出入口。

## 生产限制

- PM2 使用 `exec_mode: fork`、`instances: 1`，不要启用 cluster 或多节点写入。
- 不使用 `npm run dev` 作为赛事入口；它不提供共享评分服务。
- 对公网开放时通过反向代理、HTTPS、安全组或 VPN 限制访问；中国大陆域名访问按服务商和监管要求完成备案。
