# 管理员统一控制与后台化方案边界研究

日期：2026-07-13

## 研究目的

本研究检验以下方案是否足以支撑决赛现场：管理员统一派发当前评分队伍、管理员维护队伍和评委账号、管理员控制成绩展示，以及评委端只做受控录分。

结论：方向正确，但不能只把现有管理员长页面改成后台样式。队伍、账号、评分名册、当前派发和展示发布必须成为服务端的一等数据；否则新增评委、切换队伍、投屏和延迟请求会改变成绩口径或产生错误写入。

## 现状与缺口

- [shared/contestData.js](../shared/contestData.js) 在源码中固定生成 4 个组别、80 支队伍和固定编号；排序校验也只接受这些固定编号。当前“队伍信息编辑”只是名称/项目覆盖，不能新增真正的队伍实体。
- [contest-server.mjs](../contest-server.mjs) 中账号、密码和 7 位评委 ID 固定在内存常量；无法安全地新增、禁用或重置评委账号。
- 服务端只有 `activeGroupId`，没有“当前评分队伍”或“当前展示队伍”。评委写分只校验组别，不能证明该分数属于管理员当前派发的队伍。
- 当前成绩展示与评分目标没有独立状态。现场需要允许“评委已开始下一队，投屏仍展示上一队最终成绩”，因此两个目标不能复用同一个字段。

## 外部依据

- [OWASP Authorization Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Authorization_Cheat_Sheet.html)：建议最小权限、默认拒绝，并在每个请求上校验授权。因此“评委页面没有队伍选择器”不够，服务端也必须拒绝未派发队伍的写入。
- [RFC 9110 Conditional Requests](https://www.rfc-editor.org/rfc/rfc9110.html#section-13.1)：状态变更请求可使用前置条件和失败响应防止客户端基于过期状态覆盖更新。现有 `serverRevision` 可保留，但还应加入派发版本校验。
- [MySQL InnoDB ACID Model](https://dev.mysql.com/doc/refman/8.0/en/mysql-acid.html) 和 [FOREIGN KEY Constraints](https://dev.mysql.com/doc/refman/8.0/en/create-table-foreign-keys.html)：相关状态应在一个事务中提交，并用关系约束保护评分记录引用的队伍和评委。
- [OWASP Password Storage Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html)：密码不能以明文保存，应使用慢速、带唯一盐的密码哈希。
- [NIST SP 800-63B](https://pages.nist.gov/800-63-4/sp800-63b/authenticators/#passwordver)：密码验证方应限制连续失败登录尝试，并对会话设置明确的重新认证/超时边界。
- [OWASP Logging Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Logging_Cheat_Sheet.html)：应保留数据新增、修改、删除等可重建操作顺序的审计轨迹，同时避免记录敏感会话标识和值。
- [MDN AbortController](https://developer.mozilla.org/en-US/docs/Web/API/AbortController) 和 [MDN VisualViewport resize](https://developer.mozilla.org/en-US/docs/Web/API/VisualViewport/resize_event)：前者可取消已经失效的异步请求；后者可响应可视视口变化，用于让固定评分键盘不遮挡当前评分项。
- [MDN Server-Sent Events](https://developer.mozilla.org/en-US/docs/Web/API/Server-sent_events/Using_server-sent_events)：SSE 适合服务器向多个评委端单向推送“当前派发已变化”；它可作为轮询的加速层，但不是正确性的唯一保障。

## 建议的领域模型

### 队伍

`team` 应成为持久化实体，而不是静态候选覆盖：

```text
id                稳定内部 ID，评分后永不改变
groupId           固定组别，第一期保持现有四组
teamName          队伍展示名
projectName       项目/产品展示名
appearanceOrder   组内出场顺序
status            active | withdrawn | archived
revision          乐观并发版本
```

有评分、已派发或已展示历史的队伍只能停用/退赛，不能硬删除。新增队伍默认加入所选组别末尾；调整顺序不改变其 `id` 或任何历史评分归属。

### 账号与评分名册

`account` 应独立持久化：

```text
id, username, displayName, role, status,
passwordHash, passwordVersion, authVersion, createdAt, updatedAt
```

- 角色第一期只需 `admin` 和 `judge`；管理员管理评委，不开放自助注册。
- 密码只在创建/重置时输入；数据库只保存带盐的慢速哈希，日志和 API 响应不返回密码或哈希。
- 用户名全局唯一；不能禁用或删除最后一个启用的管理员。
- 账号禁用、密码重置或管理员强制退出时递增 `authVersion`，后续请求立即判定旧会话失效；同账号多设备登录本身仍允许。
- 登录失败要限速并写入脱敏审计。现场默认 7 位评委可作为初始化名册，但不能继续由源码常量定义。

评分名册必须与账号列表分开。推荐规则是：管理员在开赛前配置有效评委，首次派发队伍后锁定本场名册。这样新增账号不会反向改变已经完成队伍的“去高去低”分母和最终分。

若必须在比赛中增减有效评委，应为每支首次派发的队伍保存 `judgeRosterSnapshot`，并在管理员强制变更时记录原因、影响范围和重新发布操作。该能力明显增加规则与测试复杂度，不建议作为第一期默认流程。

### 控制状态

应保存两个独立的服务端控制对象：

```text
activeAssignment
  groupId, teamId | null, status, assignmentRevision, rosterSnapshot, updatedAt

displaySelection
  teamId | null, publicationStatus, displayRevision, publishedAt, updatedAt
```

`activeAssignment` 驱动评委端，`displaySelection` 驱动投影页。二者各有版本号，管理员可在下一队评分期间继续展示上一队的最终结果。

派发状态建议为：`idle -> scoring -> awaiting_submissions -> final -> closed`。没有派发时，评委端显示等待管理员安排，不能打开评分键盘。

## 必须写入方案的操作规则

1. 管理员派发队伍时，服务端在同一事务中校验组别、名册、上一个队伍状态并递增 `assignmentRevision`。不能先写组别、再写队伍。
2. 评委每次保存或提交分数都带上当前 `assignmentRevision` 和条目版本。服务端同时验证：账号角色、账号状态、评委身份、当前派发队伍、派发版本、条目版本。
3. 派发发生变化时，客户端取消旧派发尚未完成的请求，清空旧派发的自动补交队列；服务端仍必须拒绝随后到达的旧版本请求。客户端不能因为轮询较慢而继续向旧队伍写入。
4. 正常情况下，只有上一个队伍达到本场名册的全部提交后才能派发下一队。若管理员必须中断，使用单独的“强制切换”动作，要求原因并记录审计；不能静默切换。
5. 管理员撤回某评委的已提交分数后，若该队已发布到大屏，结果状态立即改为“待复核”，大屏不再继续显示旧的最终分。重新达到最终条件后，管理员显式重新发布。
6. 投影默认只允许发布最终结果。若要展示至少 3 人产生的暂算分，必须是管理员明确触发的临时发布，不应自动覆盖大屏。
7. 同分最高/最低分仍按“各去掉一条记录”计算；数值相同不影响最终平均，但规则和展示文案要说明这一点。
8. 评分细则第一期保持固定，不在后台开放修改。否则需要对每份历史评分绑定细则版本，不能与队伍/账号管理一起隐式上线。

## 边界场景与预期行为

| 场景 | 必须行为 | 失败时不可接受的结果 |
| --- | --- | --- |
| 管理员尚未派发队伍 | 评委端只显示等待状态，评分和提交禁用 | 默认打开第一支队伍并可录分 |
| 管理员在评委录入时切换队伍 | 正常切换应被完成度拦截；强制切换要记录原因，旧请求按派发版本拒绝 | 延迟请求写到新队或自动重试到错误队 |
| 评委离线后恢复 | 仅在派发版本仍匹配时补交；不匹配时保留待处理草稿并提示管理员 | 静默丢分或把旧草稿写入新队 |
| 同一评委在两台设备登录 | 保持多设备会话，但同一评分条目使用版本冲突保护；后台可显示设备冲突提示 | 两端最后写入者无提示地覆盖另一端 |
| 新增或停用评委 | 开赛前允许修改名册；开赛后默认不改变有效名册 | 已完成队伍的提交数、极值和综合分被重新解释 |
| 评委临时替换 | 保留原账号和历史，管理员创建/启用替代账号并按明确规则加入后续名册 | 删除原评委并破坏历史外键/审计 |
| 队伍退赛或编辑名称 | 使用 `withdrawn` 或 `archived`；名称/项目可改，稳定 ID 不变 | 删除带评分队伍或让排序把分数错配 |
| 出场顺序在评分中改变 | 只影响后续队列；当前 `teamId` 不变 | 当前评分页面因为序号变化跳到另一支队伍 |
| 已发布大屏结果被撤回 | 标记待复核并停止/替换展示，完成后管理员重新发布 | 大屏继续显示已失效的最终分 |
| 评委少于 3 人 | 不计算去高去低综合分，也不能标记最终 | 除以空集合或把暂算分当最终分 |
| 登录暴力尝试 | 按账号限速、记录脱敏登录失败、允许管理员恢复 | 明文密码、无限重试或日志泄露 token |
| 长队名、横竖屏、分屏或视口缩小 | 键盘打开/切项时重新计算可见区域，当前评分项完整可见 | 固定键盘覆盖当前输入项或遮住提交区 |
| 用户偏好减少动效 | 键盘定位仍发生，但不使用明显回弹动画 | 为了动画牺牲可操作性或无视系统偏好 |
| MySQL 写入中断 | 控制状态、队伍状态和关联操作使用事务；评分保留版本与审计 | 组别已换但队伍未派发，或展示指向不存在队伍 |

## 后台信息架构建议

后台默认进入“赛事控制”，而不是把所有管理功能堆在一个长页面中：

```text
赛事控制     当前派发、7 位提交矩阵、强制切换、应急操作
队伍管理     列表、搜索、创建、编辑、退赛/归档、出场顺序
评委管理     账号创建、状态、密码重置、有效名册
成绩展示     选择并发布投屏队伍，打开受控投影窗口
操作记录     只读审计筛选（可作为第二期）
```

管理员控制区应是桌面优先的紧凑表格与详情面板；评委端保持平板优先。不要把评委页面的宽松卡片布局直接放大成后台，也不要让投影页承担管理菜单。

## 键盘回弹的实现边界

“主动回弹”建议定义为功能性定位，而不是装饰动画：

1. 选择评分项、上一项、下一项时，在布局完成后读取评分行和自定义键盘的实际矩形。
2. 当前评分行若被键盘遮挡，则将页面滚动到该行完整露出的位置；在 `visualViewport.resize`、`scroll`、方向变化时再次校正。
3. 为评分行使用 ref，而不是全局 `querySelector`；为当前派发变化和页面卸载清理监听器与动画帧。
4. 使用短暂、可取消的底部进入过渡；`prefers-reduced-motion` 下保留定位但关闭动画。
5. 不使用原生输入框作为主控件，不引入浏览器数字键盘；现有自定义 0.01 键盘、退格、清空和收起行为保持不变。

## 分期建议

### 第一阶段：数据和权限基础

- 设计并迁移 `teams`、`accounts`、有效评委名册与控制状态。
- 将现有 80 支队伍、7 位评委、评分、出场顺序和展示覆盖完整迁入新模型。
- 账号哈希、登录限速、最后管理员保护、软删除规则和审计字段上线。
- 文件模式和 MySQL 模式都完成迁移、导入、导出、清场与回滚演练。

### 第二阶段：现场控制正确性

- 新增管理员派发、关闭、强制切换和展示发布 API。
- 使用资源版本/前置条件、服务端授权、事务和客户端请求取消保证不会错队写分。
- 将轮询保留为兜底；如现场需要更快切换，再增加 SSE 推送派发/展示版本。

### 第三阶段：后台与评委界面

- 拆分管理员后台为独立模块与 URL 状态，默认“赛事控制”。
- 队伍/评委管理页完成 CRUD 的安全子集和状态提示。
- 评委端删除队伍自主导航，接入等待、派发、完成、待复核状态。
- 投影页改为被管理员控制的只读消费者；完成键盘回弹与横竖屏验证。

### 第四阶段：现场回归

- 覆盖 7 台评委、管理员双端、投影端、同账号双设备、延迟/断网/恢复、强制切换、撤回再发布、队伍退赛与账号替换。
- 验证长期登录、会话失效、审计脱敏、MySQL 单行/单资源写入和数据备份恢复。

## 推荐的默认决策

1. 保持现有四个组别；第一期不开放组别或评分细则的后台编辑。
2. 有效评分名册默认为 7 位评委，并在首次派发后锁定；新增账号不自动加入正在进行的比赛。
3. 投影页只读，选择/上一队/下一队都移到管理员“成绩展示”模块；保留 URL 只作为受控兜底，不作为公开操作面。
4. 管理员可新增队伍和评委，但有评分历史的队伍/评委一律归档，不物理删除。

这些默认决策把现场正确性放在后台便利性之前；若比赛规则要求中途改变有效评委人数或修改评分细则，应单独设计版本化规则，而不是在本次后台化改造中顺带开放。
