# 共享评分服务器审计记录

日期：2026-07-07

> 历史记录：本文是当日阶段性审计记录。当前可重复巡检以 `npm run check:daily` 为准；正式启动口径以 `docs/startup.md` 和 `docs/final-competition-runbook.md` 为准。

## 本次目标

确认每台评委平板提交到同一个局域网评分服务器，管理员看板从同一个服务器读取评委进度和队伍综合评分。

## 已实现

- 新增 `contest-server.mjs`，同时提供静态页面和评分 API。
- 新增共享服务器入口；正式开赛优先使用 `npm run contest:fresh`，带已有数据复盘时才使用 `npm run contest`。
- 服务器成绩保存到 `data/contest-state.json`。
- 服务端提供登录会话，`/api/state`、`/api/entries/*`、`/api/candidates/*` 已按角色校验；未登录请求会被拒绝。
- 评委只能写入自己的评分，管理员才能编辑队伍信息和执行应急写入。
- 成绩展示页改用 `/api/scoreboard` 只读摘要接口，不会把本机缓存反向写回服务器。
- 管理员看板每 2 秒刷新服务器状态。
- 评委端每 5 秒刷新服务器状态并自动补交本机未同步的新分数。
- 评委端和管理员端都显示服务器连接状态。
- 每个角色都有退出入口。
- 登录页账号卡只填入账号，不显示或填入密码，不再绕过登录按钮。
- 服务端使用 `serverRevision` 做权威版本号，不依赖各平板系统时间判断新旧写入。
- 前端按“评委 + 队伍”串行保存，快速连续输入不会因为请求乱序丢掉最后一次有效分数。
- 服务端发现成绩文件损坏时会拒绝读写，避免把正式成绩静默覆盖为空数据。

## 代码审查发现与处理

- 离线提交不会自动补交：已处理。评委端现在轮询服务器，重连后会合并本机新分数并自动补交。
- 评委端刷新可能用服务器旧数据覆盖本机暂存：已处理。只有在同一个 `serverRevision` 下，本机 `clientUpdatedAt` 更新时才保留本机版本。
- 设备时间不一致会影响新旧判断：已处理。服务端用 `serverRevision` 判断是否允许覆盖。
- 服务端拒绝旧写入但前端仍显示成功：已处理。前端会用服务端返回的权威 entry 校准页面，并在冲突时提示重新核对。
- 管理员撤回或清空不会同步到评委端：已处理。评委端定时刷新服务器状态。
- 登录账号卡绕过密码：已处理。账号卡现在只填入账号，不显示或填入密码，仍需手工输入密码并点击登录。
- 状态文件损坏后静默变空：已处理。服务器现在报错并拒绝继续写入。
- 登录和写入完全依赖前端：已处理。当前写 API 需要服务端会话和角色权限。
- 管理员页或成绩展示页可能把本机缓存写回服务器：已处理。只有当前登录评委页允许补交自己的 pending 评分；管理员和成绩展示页只读刷新。
- 单队成绩展示页依赖外部 Cloudinary 视频和 Google Fonts：已处理。当前展示页仅使用同源静态背景和系统字体，并保持 16:9 stage。
- 管理员撤回提交一键生效：已处理。撤回提交保持单击生效。

## 仍需现场管理

- 管理员和评委账号允许多台设备同时登录；同账号再次登录不会让旧设备会话失效。
- 当前持久化仍是单 JSON 文件，生产必须单实例运行；如需多实例云部署，应先迁移到数据库或增加跨进程锁与操作日志。
- 管理员编辑队伍显示信息已同步到服务器；离线状态下会本机暂存，恢复连接后需核对同步结果。
- 即使已有服务端会话，正式现场仍必须使用专用局域网或受控云访问入口，避免无关设备接入。
- 决赛不要使用 `npm run dev`；正式清场启动使用 `npm run contest:fresh`。
